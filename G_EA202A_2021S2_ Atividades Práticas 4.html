Atividades Práticas 4

Docker

Virtualização é o processo de criação de uma representação com base em software, ou virtual, de algum tipo de recurso ou aplicação como aplicativos virtuais, servidores, armazenamento e redes. Essa é a maneira mais eficaz de reduzir as despesas de TI e, ao mesmo tempo, aumentar a eficiência e a agilidade para empresas de todos os portes.

Nesse contexto o Docker se apresenta como uma ferramenta feita para facilitar a criação, deployment, e execução de aplicações ao utilizar containers. O uso de containers permitem que um desenvolvedor "empacote" sua aplicação com todas as partes que ela necessita, como bibliotecas e outras dependências, e lance como um único pacote. Ao fazer isso, graças ao container, pode-se assumir tranquilamente que a aplicação irá executar corretamente em qualquer outro sistema Linux independentemente de configuração personalizada que possa diferir das máquinas usadas para implementação e teste.

De certa forma, Docker funciona de forma similar a uma máquina virtual. Mas, diferente de uma máquina virtual, ao invés de criar um sistema operacional virtual completo, o Docker permite que as aplicações utilizem o mesmo kernel Linux do sistema no qual estão executando e exige apenas que tais aplicações sejam empacotadas juntamente com os requisitos que não são encontradas no host.  Isso fornece um aumento significativo de performance e reduz o tamanho das aplicações.

Instalando o Docker na sua Máquina

Para que você possa utilizar o Docker, antes de mais nada, é necessário que o Docker esteja instalado na sua máquina. Existem versões do Docker para ambiente Linux, Windows e Mac. Se o seu ambiente é Windows, você tanto pode usar imagens Linux como imagens Windows. Em nosso curso, sempre utilizaremos imagens Linux. Dessa forma, quando for instalar o Docker em uma máquina Windows, tome o cuidado para instalar a versão correta. Se a sua máquina for uma máquina Windows ou Mac, você precisará também habilitar na BIOS ou UEFI do seu computador o HYPERVISOR, ou seja o suporte em hardware para o gerenciamento de máquinas virtuais. Procure informações específicas de como fazer isso na sua máquina, antes de instalar o Docker Desktop. Você precisará rebootar sua máquina para fazer isso, pois a configuração do hypervisor é feita nas configurações de boot da máquina, antes do carregamento do sistema operacional.  

Ainda, se você usa o ambiente Windows 10, sugerimos instalar o Ubuntu for Windows e o WSL2: Windows Subsystem for Linux 2, ferramentas que permitem executar um Kernel Linux nativo, em paralelo com o próprio Windows 10.  Depois que instalar o Ubuntu for Windows, você poderá habilitar, no Docker Desktop, em Settings -> Resources -> WSL Integration, a integração do Docker Desktop com o Ubuntu for Windows, de tal forma que, instalando o docker-ce no Ubuntu, ele poderá utilizar a mesma imagem instalada no Windows 10 e compartilhar as imagens baixadas no Windows 10. 

Para instalar o docker-ce no Ubuntu, é necessário antes adicionar o repositório docker:

sudo apt-get update
sudo apt-get install apt-transport-https ca-certificates curl software-properties-common
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo apt-key add -
sudo add-apt-repository "deb [arch=amd64] https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable"

E então, para instalar o Docker CE

sudo apt-get update
sudo apt-get install docker-ce

Executando Um Container

Uma vez que o Docker esteja instalado na sua máquina, podemos executar nosso primeiro container. Para isso, é necessário que escolhamos um container para executar. Podemos tanto utilizar um container construído por terceiros, disponibilizado publicamente em um repositório central, chamado de DockerHub, ou podemos utilizar um container que nós mesmos construímos. Vamos inicialmente executar um container de terceiros, no caso o container gerado pelos desenvolvedores do Node.js:

Execute os seguintes comandos:

docker pull node
docker run -it node bash

 O resultado deverá ser algo parecido com isso (o número será diferente):

root@04bae469eec0:/#

Isso significa que você está rodando um command-shell bash dentro do container node.js. Daí para frente, você poderá utilizar comandos Unix para se movimentar dentro do container, e utilizar o comando node para executar programas desenvolvidos em JavaScript. Por exemplo, se você criar um programa com o nome prog.js, você poderá executá-lo chamando “node prog.js”.  

Dentre os comandos “docker” que você executou, o primeiro comando irá procurar no DockerHub pela imagem docker mais recente e fazer o download enquanto o segundo comando irá executar o container  em modo interativo (argumento -it). O programa docker possui um CLI (Command Line Interface) com um grande número de parâmetros, cujo significado pode ser consultado por meio da documentação referenciada no link acima. Esse CLI se assemelha muito com comandos do UNIX. Dessa forma, se você conhece um pouco do UNIX, pode se beneficiar dessa similaridade. Se você digitar, por exemplo, somente “docker”, ele mostrará para você todas as possibilidades de sub-comandos possíveis. Assim, você descobrirá que existem comandos como “docker ps”, que mostra quais os container que estão ativos, “docker images”, para saber quais as imagens que já foram carregadas na sua máquina, “docker rm” para apagar um container que foi criado e “docker rmi” para apagar uma imagem que foi criada. 

Dois comandos são de especial importância. O “docker run” e o “docker exec”. Todo container possui uma imagem. Essa imagem é que é baixada do DockerHub. Antes que um container possa ser utilizado, é necessário que criemos uma instância do container, Para isso, utilizamos o “docker run”. Quando executamos o comando acima, criamos uma instância do container e o executamos.  

Porém, nem todo container precisa ser executado em modo interativo (ou seja, com as flags “-it”. É possível executar containers que ficam rodando em background. Nesse caso, quando queremos ter acesso ao container, utilizamos então o comando “docker exec”. Nesse caso, podemos fazer um “docker exec -it <nome container> bash” e ter acesso ao container. Da mesma forma, se queremos ter múltiplos acessos a um mesmo container, isso é possível com o “docker exec”. 

Observe que o container que criamos é um container Linux. Se você não está acostumado com o mundo Unix, provavelmente se sentirá um “peixe fora d'água”. Se você já conhece o Unix, então provavelmente já está habituado. Se não conhece,  utilize os comandos “ls” para listar o diretório e “cd” para mover o diretório, e explore um pouco o diretório interno do container do node.js que criamos e tente entender como a árvore de diretório do container se relaciona com a árvore de diretório do seu disco externo. Você descobrirá que elas são completamente independentes. Nesse caso, ficará uma dúvida: como fazer para comunicar os dois mundos, o mundo de dentro do container com o mundo de fora do container ? 

Controlando o Entorno do Container

Você já descobriu que o Container é como uma caixa isolada, como se fosse uma máquina separada da sua própria máquina. Ela pode, inclusive, rodar um sistema operacional diferente. Como fazer a comunicação entre elas ? Para isso, há uma série de configurações possíveis que podem ser realizadas. 

A opção “-v” permite que um volume do seu computador seja montado internamente dentro do container. Com isso, é possível que você passe arquivos para dentro do container. Com isso, você pode enviar programas, arquivos, configurações, etc, para dentro do container, e utilizá-los de dentro do container. Esse é o mecanismo padrão, para que você possa utilizar um container, sem que você precise construir um container próprio, podendo reutilizar containers de terceiros. 

A opção “-p” permite que você exporte portas de rede internas do container para a sua máquina. Dessa forma, por exemplo, se você cria um programa em node.js que usa a porta 80, dentro do container, e quer que a sua máquina escute nessa porta, você pode fazer isso usando essa opção. Você pode, inclusive redirecionar para outra porta, caso a porta em questão esteja sendo usada por outra aplicação. 

A opção “--name” permite que você customize o nome do container. Caso você não dê um nome ao container, o docker criará um nome para ele. Você verá que o docker utilizará nomes bastante estranhos para batizar os containers, que darão um certo trabalho para você remover depois. Recomendamos que você sempre dê nome aos seus containers, para facilitar o processo de remoção, depois. 

A opção “-rm” faz com que, ao terminar um container, ele automaticamente seja destruído. Caso contrário, o docker assume que você terá interesse em re-startá-lo posteriormente. Nesse caso, ao invés de re-iniciá-lo com um docker run, você precisará dar um “docker start” ao invés de “docker run”. Para destruir de vez um container que você não quer mais, use o “docker stop”. 

A melhor opção para iniciar um container é criar um pequeno script em bash, contendo todos os comandos necessários, de acordo com suas necessidades, e deixá-lo à mão, para quando precisar. Nossa sugestão é, inclusive, que você deixe as opções separadas linha a linha, para facilitar sua compreensão, e para alterá-lo facilmente, caso seja necessário. Dessa forma, uma possível versão do comando para execução do node.js poderia ser algo como:

#! /bin/bash
SCRIPTPATH="$( cd "$(dirname "$0")" ; pwd -P )"
docker run \
-it \
--name node.js \
--rm \
-d \
-h node.js \
-w /node \
-v ${SCRIPTPATH}/:/node/ \
-p 80:80 \
node

Copie esse script em um arquivo bash e o execute (você precisará ter o bash instalado na sua máquina para isso funcionar). 

Esse script, quando executado, cria e executa uma instância do container node chamada node.js, deixando-a em background. Veja no manual do docker o significado de cada um dos parâmetros que utilizamos no script para entender exatamente o que ele faz. 

Essa instância do container pode ser acessada a qualquer momento por meio do docker exec:

docker exec -it node.js bash

Da mesma forma, quando dermos exit, o container continuará sendo executado, pois o container original foi executado a partir do script. Para encerrar definitivamente o container, precisaremos fazer um:

docker stop node.js

Criando Novos Containers a Partir de Containers Pré-Existentes

A partir de um container em que você esteja executando, é possível aumentá-lo e com isso criar um novo container. Por exemplo, se você tem um container que foi criado a partir de uma imagem do Ubuntu, você pode, por exemplo, fazer um update da imagem e começar a fazer atualizações nele:

apt update
apt install -y curl

Para fazer atualizações no container, você deve fazer:

docker container commit CONTAINER_ID
docker image tag <IMAGE_ID> <NAME>

onde CONTAINER_ID é o id do container onde as mudanças foram feitas, <IMAGE_ID> é o id da imagem criada quando o comando anterior é executado e <NAME> é o nome que será dado a esta imagem.

Criando uma imagem a partir de um dockerfile

Pode-se ainda criar uma imagem a partir de um conjunto de diretrizes de construção, um dockerfile.  Apesar do resultado final ser essencialmente o mesmo (uma imagem personalizada), o uso de um dockerfile  permite que o usuário seja capaz de fazer uma melhor gestão de mudanças, especialmente conforme as imagens vão ficando maiores e mais complexas. Um exemplo disso é quando um programa recebe uma atualização: a imagem gerada via docker commit  precisaria ser gerada novamente, seguindo todos os passos já executados, enquanto com o dockerfile  seria necessário apenas recompor a imagem, uma vez que este arquivo contém os comandos apt  para instalar os programas.

Um dockerfile  exemplo é mostrado abaixo

FROM alpine
RUN apk update && apk add nodejs
COPY . /app
WORKDIR /app
CMD ["node","hello.js"]

No exemplo dado, uma vez que seja dado o comando de build  o docker irá buscar uma imagem chamada "alpine", irá atualizar os pacotes e instalar o "node.js", copiará todo o conteúdo do diretório atual da máquina host  para uma pasta chamada "app" dentro da imagem, entrará na pasta recém criada e executará um programa chamado "hello.js" (supondo que o mesmo estivesse na pasta do dockerfile).

Os comandos de um dockerfile podem ser encontrados em mais detalhes no site do docker.

Deliveries

Nosso intuito é usar o Docker para criar containers com o software necessário para os pacotes de software que precisaremos para os desenvolvimentos que utilizaremos em nosso curso. Em nosso curso, utilizaremos o stack MEVN, que inclui o MongoDB, o Express, o Vue e o Node.JS. Na aula de hoje, começaremos nosso estudo utilizando o Node.js. Dessa forma, nosso objetivo é aprender a usar um container com o node.js e fazer uso de suas funcionalidades, de forma que nas próximas aulas, possamos utilizá-lo de maneira prática. Nosso maior objetivo aqui é capacitá-lo para que você possa, a partir da próxima aula, estar com todas as ferramentas necessárias para iniciar o desenvolvimento de um aplicativo MEVN. Para isso, diversos passos são necessários, tais como ter o docker funcional, ter o Visual Studio Code funcional, ter o Node.js funcional e saber executar um programa com o node.js rodando por meio de um container docker. Dessa forma, o maior desafio é realizar a instalação de todas essas ferramentas em sua máquina e deixar o ambiente todo preparado para que o desenvolvimento possa ocorrer sem maiores problemas. Os deliveries que vamos solicitar aqui são na verdade mais um “landmark” para testar que tudo isso está funcionando adequadamente, do que de fato o objetivo final do trabalho. Pense nisso quando estiver preparando seu delivery. 

Delivery 1

Como primeira atividade prática de hoje, o desafio é o seguinte. Estude como criar um ambiente em que um container docker é usado para “baixar” o Node.js (ou seja, não precisaremos baixar de fato o node.js, porque o docker fará isso para a gente automaticamente), e possamos utilizá-lo para fazer o deployment de nossa aplicação em JavaScript, desenvolvida sob Node.js. Para isso, precisaremos criar um container com o node.js, passar para o container um diretório contendo um programa em JavaScript, de tal forma que esse programa seja executado e possa ter sua funcionalidade utilizada. No tutorial do w3schools sobre Node.js, temos um demo mostrando como criar um programa em JavaScript que funciona como um Hello World, onde o programa publica na porta 8080 de localhost a famosa frase “Hello World”. Em nosso desafio número 1, queremos que você gere um script em bash, que execute um container com o node.js, que por sua vez deve executar um programa, inspirado no demo do Hello World do exemplo do w3schools, mas para publicar a famosa frase, entretanto, na porta 2020 de localhost, que deve ser aberta em um browser. 

Você deve disponibilizar esse script no gitlab, na pasta aula4 do repositório deliveries, usando um merge-request, da mesma forma que foi feito nas aulas 1 e aula 3. 

Passos para a resolução do exercício:

- Se for ambiente Windows {
    - Instalação do Ubuntu for Windows
    - Instalação do WSL: Windows Subsystem for Linux
    - Conversão do WSL para WSL2
}
- Instalação do Docker
- Se for ambiente Windows {
    - Habilitar Integração WSL no Docker para Ubuntu
}
- Execução com sucesso do container node.js
- Executar com sucesso a passagem de arquivos para dentro do container
- Executar com sucesso a execução de um programa JavaScript de dentro do container
- Liberar com sucesso uma porta TCP-IP para comunicação com o container
- Execução com sucesso do programa Hello World
- Empacotamento com sucesso de tudo isso dentro do script
- Disponibilização com sucesso de tudo isso no repositório

Delivery 2

Agora que você criou a primeira aplicação em JavaScript desenvolvida sob o Node.js, vamos explorar um pouco o chamado NPM (Node Package Manager) que é um gerenciador de pacotes para pacotes node.js. Em www.npmjs.com podem ser encontrados vários pacotes gratuitos para baixar e usar. O programa NPM já é instalado no seu computador quando você executa o container com o Node.js. 

Descubra como fazer o download/instalação do package “upper-case” usando o npm. 

Quando você instala um pacote com o npm, ele automaticamente acomoda esses pacotes, chamados de modules, em um subdiretório chamado de “node_modules”, que é o diretório onde o node espera encontrar todos os modules necessários para a execução. Dessa forma, se você estiver trabalhando em um ambiente Windows, e seu projeto estiver localizado em C:\Users\SeuNome\SeuProjeto, para a execução do pacote upper-case em seu projeto, a estrutura deve ficar parecida com C:\Users\SeuNome\SeuProjeto\node_modules\upper-case

Agora, a proposta é você usar essa função do upper-case no script do Hello World, que utilizamos no Delivery 1, para que mostre a frase toda em caixa alta no browser. De fato, isso significará apenas acrescentar uma única linha de código, ao código original utilizado no Delivery 1. 

Assim, como Delivery 2, acrescente o uso do module upper-case para criar um Hello World em caixa alta que utilize esse module e faça um commit com a alteração no gitlab, na pasta aula4 do repositório deliveries e utilize um merge-request. Não vale fazer o Hello World em caixa alta em hard-code: HELLO WORLD, heim ! O exercício é usar o npm para aprender a usar o mecanismo de package manager, e não gerar o hello world em caixa alta !
