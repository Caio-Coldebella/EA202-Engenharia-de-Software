Atividades Práticas 5

O Framework MEVN

Em nosso curso, vamos desenvolver uma aplicação distribuída, empregando a arquitetura 3-Tier segundo o estilo REST/HTTP (Representational State Transfer) e implementada em Javascript sobre o framework MEVN (MongoDB-Express-Vue-Node.js). O framework MEVN é o que chamamos de um stack, sendo constituído por um frontend, ou camada do cliente, que normalmente roda em um browser ou em um app de smartphone, um backend, ou camada do servidor, que roda em um servidor remoto, em um servidor de cloud, por exemplo, e que usualmente ainda utiliza os serviços de uma camada de serviços de retaguarda, normalmente um banco de dados ou outro tipo de persistência de dados. Existem diversos frameworks que têm se tornado bastante populares para uma implementação rápida e eficiente de aplicações distribuídas, nas mais diversas linguagens. Para a linguagem Javascript, os frameworks de frontend mais populares vêm sendo o Angular, o React e o Vue. O MEVN é somente um dentre três frameworks que se popularizaram, o MEAN, o MERN e o MEVN, cada um deles usando uma diferente tecnologia de frontend. Dentre esses três (Angular, React e Vue), o Vue é o que possui a menor curva de aprendizado. Dessa forma é o que usaremos em nosso curso. 

Veja o significado de alguns dos termos em negrito abordados acima:

Aplicação distribuída: uma aplicação distribuída possui diversos componentes distribuídos ao longo de uma rede. Seus componentes executam em diferentes nós de processamento dispersos em diferentes domínios administrativos.

Arquitetura 3-Tier: os componentes da arquitetura executam em 3 diferentes camadas horizontais (chamadas de tiers): cliente, servidor e serviços de retaguarda ou suporte.



Estilo REST/HTTP (RESTful): os métodos GET, POST, PUT e DELETE do protocolo HTTP são utilizados para acessar, inserir, modificar e remover informações no domínio do servidor.

REST (Representational State Transfer) é um estilo arquitetural proposto por Roy Fielding em 2000 (tendo tornado-se extremamente popular quase uma década depois).

REST define recursos identificados por URI/URL e mantidos em servidores stateless. Cada recurso tem um estado associado que pode se alterar ao longo  do tempo. REST prevê ainda que a arquitetura é organizada em camadas horizontais (tiers).
Arquiteturas que seguem o estilo REST são comumente implementadas em 3 camadas (3-tier) com interação cliente-servidor via HTTP. As seguintes requisições HTTP são empregadas:
POST: cria um recurso com seu respectivo estado inicial no servidor;
GET: acessa o estado de um recurso mantido no servidor;
PUT: altera o estado de um recurso mantido no servidor;
DELETE: remove um objeto mantido no servidor.

Normalmente o estado dos objetos é persistente em banco de dados. Operações HTTP POST, GET, PUT e DELETE são mapeadas em operações CRUD (Create, Retrieve, Update, Remove) no banco de dados.

Framework: esqueleto de uma aplicação onde diversos componentes já estão implementados.

O Framework MEVN integra diversas tecnologias para a construção de uma aplicação Web: MongoDB, Express, VueJS e NodeJS.

Na aula de hoje, vamos consolidar os conhecimentos já exercitados informalmente na aula passada (exercício para casa) sobre o Node.js e o Express, que devem implementar a camada do servidor da arquitetura 3-Tier e também do MongoDB, nosso banco de dados disponível na camada de serviços de retaguarda.

Introdução ao Express

Como vimos na aula passada, o Node.js é um interpretador JavaScript que pode ser utilizado para implementar programas em JavaScript em computadores quaisquer, não necessariamente rodando dentro de um Browser. Uma das utilizações do Node.js é a construção de servidores de Backend, para atender a requisições vindas de páginas rodando em aplicações web, tanto em páginas web como em aplicações em smartphones. Uma das formas de construir esses servidores é utilizando o npm para agregar diversos pacotes diferentes em JavaScript e construir programas mais sofisticados. Um dos pacotes JavaScript mais utilizados para essa finalidade é o Express. O Express é um pacote JavaScript que provê funcionalidades REST, ou seja, permite a construção de pontos de acesso do tipo GET, POST, PUT e DELETE. 

Estrutura de um Servidor Express rodando sobre Node.js

O REST é um estilo arquitetural construído sobre o protocolo HTTP (Hyper Text Transfer Protocol). O protocolo HTTP especifica um conjunto de operações, também chamadas de "verbos", realizadas por meio de sockets. Apesar de existir um grande número de verbos HTTP, os mais comuns são o GET, o POST, o PUT e o DELETE. Esses verbos correspondem a operações aplicados sobre um recurso, especificado por meio de uma URL, ou Universal Resource Locator, que pode conter variáveis. Uma URL como por exemplo /alunos define um recurso sem variáveis, que pode ser acessado por diversos verbos. Uma URL como por exemplo /alunos/:ra define um recurso onde :ra é uma variável, podendo se referir a diversos recursos diferentes. Por exemplo, o recurso /alunos/123456 é uma instância de /alunos/:ra onde :ra é igual a 123456. Para cada instância de recurso, diferentes verbos HTTP podem ser programados e portanto é necessário definir um código para cada operação diferente. O retorno enviado pelo servidor a uma requisição HTTP é chamado algumas vezes de payload e corresponde a uma resposta à requisição. Em princípio, esse payload deve seguir um formato padronizado, também chamado de tipo MIME (Multipurpose Internet Mail Extensions), que é o mesmo utilizado para formatar extensões de mensagens de e-mail. Um tipo MIME muito utilizado é o application/json que define que o retorno é um objeto JSON, ou JavaScript Object Notation. A vantagem de usar o JSON é que ele pode ser utilizado muito facilmente em scripts em JavaScript, sendo interpretados automaticamente como o valor de uma variável em JavaScript. Alguns verbos, como por exemplo o POST e o PUT podem demandar ainda um parâmetro que é o próprio recurso sendo enviado, também em formato JSON. Processamento HTTP no Servidor

O Express é na verdade tanto uma biblioteca de funções JavaScript e um CLI, ou Command Line Interface, um comando que executa facilidades para o uso da biblioteca. Como biblioteca de funções JavaScript, o Express permite que se definam facilmente rotas para o tratamento de diferentes verbos HTTP. Um código tipico utilizado no Express é como o código a seguir:

var express = require('express');
var router = express.Router();

router.route('/alunos')
.get(function(req, res)
         {
         // se a requisição for get
         ...
         }
     )

.post(function(req, res)
         {
         // se a requisição for post
         ...
         }
     );

Observe que as funções de tratamento dos verbos HTTP são sempre funções de middleware. Funções de middleware são funções que necessariamente possuem os parâmetros (request, response, next), sempre nessa ordem, podendo o parâmetro next ser omitido se a função de middleware encerra a cadeia de chamadas. Os nomes dos parâmetros não precisam ser exatamente esses, mas acontecem sempre nessa ordem. O primeiro parâmetro é um objeto do tipo request, o segundo é um objeto do tipo response e o terceiro é o next. Funções de middleware têm esse nome, exatamente porque essas funções são chamadas em sequência, e assim uma função pode estar no meio de uma sequência de chamadas, chamando a próxima função da cadeia, que é sempre next. Cada função de middleware recebe um parâmetro por meio da variável request, e pode executar modificações na variável response. Caso a função de middleware não seja a última de uma sequência, ele deve necessaria chamar a função next() ao seu final, para que a cadeia de processamento de response continue. Caso ela não o faça, a sequência de chamadas é encerrada e a variável response é retornada a quem realizou a chamada. 

Veja alguns exemplos:

req: estrutura com dados da requisição
Exemplo:
req.params.ra : um parâmetro da requisição
req.body.curso : um atributo do payload (objeto JSON)
req.cookies : cookies enviados pelo cliente

res: estrutura para montar a resposta
Exemplo:
resposta = {"alunos": [dados]};
res.json(resposta) : gera um payload com um objeto JSON
res.cookie(nome, valor, opções) : adiciona um cookie na resposta

A resposta é devolvida ao cliente quando a função terminar.

Veja o conteúdo das estruturas de requisição e resposta em:

Instalando o Express

Para instalar o Express, é necessário que tenhamos uma instalação do Node.js. Em nosso caso, utilizaremos nossa versão do node.js em Docker:

#! /bin/bash
SCRIPTPATH="$( cd "$(dirname "$0")" ; pwd -P )"
docker run \
-it \
--name node.js \
--rm \
-d \
-h node.js \
-w /node \
-v ${SCRIPTPATH}/:/node/ \
-p 3000:3000 \
node

Copie o script acima para um arquivo com nome node.sh e o torne um arquivo executável

> chmod ugo+x node.sh

Dessa forma, você pode executá-lo da seguinte maneira:

> ./node.sh

A execução do script irá criar um container Docker de nome node.js com o node disponível e a porta 3000 exportada para a sua máquina. Após executá-lo, você pode ter acesso ao container executando:

docker exec -it node.js bash

e então poderemos instalar o Express:

npm install -g express express-generator 

Observe que a instalação do Express sobrevive somente enquanto o container estiver rodando. Se você fechar o container e executar seu script de novo, será necessário executar esse passo novamente. Entretanto, enquanto você mantiver o container node.js rodando em background, o Express continua lá disponível. 

Aplicação Exemplo em MEVN

Vamos iniciar a utilização do Framework MEVN (MongoDB-Express-Vue-Node.js) por meio de uma aplicação simples. Essa aplicação mantém um cadastro de alunos com as seguintes informações: RA, Nome e Curso.

Para cadastrar um aluno realizamos uma operação HTTP POST no recurso /alunos passando um objeto JSON na forma:

{"ra": "123456", "nome": "Albert Einstein", "curso": "Fisica"}

Para alterar os dados de um aluno realizamos uma operação HTTP PUT no recurso /alunos/:ra, por exemplo: /alunos/123456 e passamos um objeto JSON com os novos dados do aluno (todos os campos devem ser enviados novamente).

Para remover um aluno realizamos uma operação HTTP DELETE no recurso /alunos/:ra, por exemplo, /alunos/123456.

Finalmente, para acessar todos os alunos realizamos uma operação HTTP GET no recurso /alunos.

Executando a Aplicação

Abra um terminal no Ubuntu e crie um diretório onde disponibilizará sua aplicação, por exemplo aula5. Entre nesse diretório e crie o script com o código de execução do container, por exemplo node.sh, como sugerido anteriormente. Em seguida execute o script para subir o container, e depois utilize o docker exec -it node.js bash para ganhar acesso a ele. 

Para criar uma aplicação nova, chamada AppTeste, utilize o Express (que já deve ter sido instalado previamente no container):

> express AppTeste

O CLI do Express deve criar um diretório chamado AppTeste, que conterá um template para sua aplicação Express, com vários arquivos e subdiretórios. 

Analise o diretório AppTeste que foi criado e seus arquivos internos. Tente entender a estrutura da aplicação criada e como ela funciona, principalmente a estrutura do servidor app.js, e o arquivo routes/index.js onde as rotas devem ser inseridas. 

Em seguida, instale as dependências da aplicação

> npm install

E por fim, execute o servidor:

> npm start

Abra um browser e acesse o endereço http://localhost:3000/

Nesse ponto, você verá somente uma página com a indicação que o Express está funcionando. 

Agora precisamos inserir as rotas da nossa aplicação e construir o arquivo HTML principal da mesma:

Faça o download dos arquivos de exemplo. Mude para o diretório para AppTeste e copie para esse diretório os arquivos:

> cd AppTeste
> mv routes/index.js routes/index-orig.js
> cp ../Exemplos/index.js routes/index.js
> cp ../Exemplos/index.html  index.html

Analise o novo arquivo routes/index.js e compare com o anterior. Tente entender as inserções que foram efetuadas e por quê as mesmas foram introduzidas. 

Gere novamente a aplicação, inserindo as novas dependêncais: npm install
Execute o servidor: npm start

Abra novamente o browser no endereço http://localhost:3000/

Uma nova tela, agora como Manipulador de Recursos deve ser visualizada. O manipulador de recurso (mr) permite enviar e receber objetos JSON. Essa interface será útil para testar o futuro servidor de sua aplicação.

Testando a Aplicação

Utilizando a interface, é possível agora fazer inúmeros testes, para você entender como funciona o Express. Experimente com diferentes URLs, tentando exercitar diferentes possibilidades de acordo com o código da aplicação. Exercite a aplicação cadastrando (POST), acessando (GET), alterando (PUT) e removendo (DELETE) alunos. Verifique o que acontece, tanto na interface gráfica (no browser) como no log do node.js no container. 

Inspecione o código das rotas (routes/index.js) e na interface (index.html), e faça modificações nos mesmos, de forma a testar sua compreensão. Particularmente, no arquivo HTML, na atribuição do retorno das mensagens para a variável payload, substitua response.data por response, para ver tudo que é possível obter a partir da resposta de uma requisição HTTP. 

Introduza logs no processamento das requisições via console.log(). Aproveite para compreender o funcionamento do mecanismo REST. 

Altere o servidor para introduzir novos parâmetros na descrição do aluno, por exemplo, CR e CP.

OBS: toda a vez que houver a inserção de um novo pacote JavaScript no código do servidor, é necessário executar

> npm install

para atualizar as dependências do projeto. 

Delivery

O grande desafio agora é desenvolver o código aqui disponibilizado, para que ele se torne útil. Na aula de hoje, faremos uma primeira etapa do desafio, filtrando as informações e armazenando provisoriamente essa informação em arquivos. Em nosso exemplo, os dados enviados para o servidor de background ficam armazenados só em memória. Assim que o servidor é parado, todos os dados se perdem. A ideia então é capturar as informações recebidas, armazenar essas informações provisoriamente em arquivos, e quando elas forem consultadas, consultamos as mesmas a partir desses arquivos. Esse passo é um passo intermediário para o armazenamento em banco de dados no MongoDB, que veremos a partir da próxima aula. 

O delivery a ser encaminhado, então, corresponde à aplicação modificada, para que as informações armazenadas sejam guardadas de maneira perene em arquivos, de tal forma que quando consultadas, possam retornar os valores armazenados anteriormente. As funções POST e PUT devem atualizar esses arquivos, a função GET deve ler os dados desses arquivos e a função DELETE deve deletar os dados desses arquivos. Para testar se está funcionando, você deve parar o servidor e reiniciá-lo novamente, verificando se os dados enviados anteriormente continuam lá. 

Não se esqueça de criar um arquivo server.sh que carregue o container (o equivalente do node.sh que construímos em nosso exercício preliminar) e um start.sh que execute o servidor de dentro do container. Observe que o start.sh deve fazer mais do que simplesmente chamar o docker exec -it node.js bash, pois não queremos agora que ele só execute o bash, mas que também já chame o npm start para iniciar o servidor. Assim, ao executar o start.sh, o servidor deverá ficar completamente funcional, podendo ser acessado pelo browser. O start.sh deve ser chamado de dentro do server.sh, para que ao mesmo tempo ele inicialize o container e starte o servidor. A solução final encaminhada não deve assumir que o usuário irá executar o servidor manualmente de dentro do container, mas deve ser transparente ao usuário, que deverá apenas executar server.sh e abrir um browser no endereço http://localhost:3000

Os deliveries referentes à aplicação modificada devem ser encaminhados ao gitlab, utilizando o processo usual, no repositório deliveries, no subdiretório aula5.
