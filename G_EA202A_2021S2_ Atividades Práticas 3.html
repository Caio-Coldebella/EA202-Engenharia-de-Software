Atividades Práticas 3

HTML, CSS e Javascript

Muitos dos sistemas de software na atualidade se enquadram dentro do que poderíamos chamar de Aplicações Web. Aplicações Web são mais do que simplesmente "sites de Internet". É certo que um "site de compras na Internet" não deixa de ser um tipo muito rudimentar e tosco de uma Aplicação Web, mas o conceito de Aplicação Web é muito mais amplo do que isso, envolvendo e incluindo também, por exemplo, sistemas distribuídos com clientes que podem ser browsers de Internet, aplicativos de smartphones e servidores acoplados a bancos de dados ou provendo comunicação entre usuários em múltiplas plataformas, de maneiras mais ou menos convencionais. 

Apesar de haver várias tecnologias que podem ser usadas em aplicações Web, a tecnologia que tem sido predominante, e que estaremos estudando ao longo do curso, envolve basicamente três grandes tecnologias: HTML, CSS e Javascript. O objetivo desta aula é fazer uma introdução prática a essas três tecnologias e capacitar vocês a trabalhar com elas. 

Das três tecnologias, o HTML, ou Hyper Text Markup Language, é a mais antiga. Originalmente, o HTML era simplesmente uma linguagem de markups para hipertextos, ou seja, era realmente somente uma linguagem para a criação de páginas Web. Sua origem se dá no contexto de se consolidar uma hiper-marcação de textos, ou seja, a ideia era encontrar uma estratégia para marcar certos trechos do texto que deveriam ter um significado especial em um hipertexto. Esse significado especial poderia ser tanto, simplesmente, ser um texto de título, um marcador de capítulo ou de seção, como ser um hiperlink, um texto que quando recebesse um clique do mouse remetesse o usuário para uma outra página da Internet. Em suas primeiras versões o HTML possuía somente alguns markups, ou seja, sequências especiais, como <H1> ou <H2>, que caracterizavam o início de uma marcação e outras como </H1> e </H2> que caracterizavam o final. Esses markups eram então adicionados a texto simples, criando um texto que, adicionado de diversos markups como esse, podia ser interpretado e renderizado adequadamente em um browser de Internet. Dessa forma, o HTML era quase que um arquivo de texto com marcações adicionais para formatar títulos, hiperlinks e outras formatações especiais. Com o tempo, novos markups foram criados para adicionais cada vez mais funcionalidades, até que o HTML acabou se transformando em uma linguagem de formatação visual de páginas WEB generalizadas. Hoje com o HTML5, verdadeiras interfaces gráficas podem ser construídas com o HTML. Muitos markups, apesar de continuarem válidos, acabaram se tornando obsoletos, e sua funcionalidade foi transferida para os assim chamados CSS ou Cascade Style Sheets, ou FOLHAS de ESTILO.

A criação das folhas de estilo foi uma das grandes inovações na construção das páginas Web. Percebeu-se que o HTML deveria ser utilizado na determinação do CONTEÚDO e da ESTRUTURA da página, sendo que a formatação e o estilo, ou seja, a formatação visual, coisas como cores, fontes, estilos, tamanhos, deveriam ser determinados em outro lugar. Esse outro lugar acabou sendo as folhas de estilo. Daí nasceram as folhas de estilo ou style-sheets. Como essas folhas de estilo podiam ser aplicadas uma sobre as outras, portanto em cascata, aí ficou o nome CSS, cascade style sheets. Dessa forma, pensando que que um arquivo HTML guarda o código de uma formatação visual de uma página web, ou seja, uma página gráfica, vamos imaginar que o código HTML deve conter a estrutura e o conteúdo que deve ser renderizado. Os detalhes de formatação devem estar nas folhas de estilo, ou seja, nos CSS. 

E por fim, percebeu-se que era possível acrescentar uma dinâmica às páginas web. Botões podiam ser clicados, alguma animação podia ser realizada e para isso, então criou-se o JavaScript, uma linguagem de script inspirada na linguagem Java, que permitia que alguma pequena dinâmica fosse acrescenta às páginas Web. Posteriormente, essa linguagem acabou sendo desenvolvida e aumentada e acabou ganhando de fato as funcionalidades completas de uma linguagem de programação. Ao final do processo, terminamos com essa combinação de HTML, CSS e JavaScript, que juntos, nos permitem a criação de uma página Web dinâmica. Uma página onde é possível não somente criar elementos gráficos e de texto, mas executar desenhos, acessar widgets, elementos de formulários e operar elementos completos de um programa, transformando essa solução em uma solução completa de programação. Essa é a solução que começaremos a estudar na aula de hoje.

Siga as partes HTML5 Tutorial e HTML Forms do tutorial HTML do site  w3schools.

Deliverable

Após estudar estudar os elementos de HTML, CSS e JavaScript, a proposta é gerar uma página em HTML, na forma de três arquivos texto, um em HTML, outro em CSS e outro em JS, os três interligados, de tal forma que, quando a página HTML seja aberta em um browser, como o Google Chrome ou o Firefox, o seguinte resultado seja obtido.  

A página deve ser estruturada da seguinte maneira. Deseja-se obter um formulário estruturado em diversos campos do tipo <div>, cada um deles controlado por uma folha de estilo independente, segundo a figura 1 e as instruções a seguir. 

OBS: A Figura 1 é um esboço de como deve ser feito o exercício. Com relação ao alinhamento do formulário e ao que não está especificado como requisito, fica a critério de vocês.

1)      Cabeçalho (div1) com a logo da FEEC, clicando nela, deverá direcionar para o site da FEEC. O nome e o código da disciplina no meio.  Uma figura unificada com o símbolo do gitlab e a logo da UNICAMP, com o link direcionando para a nossa página do gitlab. O Background desse cabeçalho deverá ter a cor cinza

2)      Título: Entregável 3

3)      As páginas devem ser diagramadas com cabeçalho, coluna esquerda coluna direita e footer, tendo a região principal como pode ser vista na figura acima.

4)      Crie um formulário com os seguintes campos:

Nome*:                                                              RA*:

Data de Nascimento*:                                  Naturalidade*:

Universidade*:                                Curso*:                                Semestre*:              

  • Os itens do formulário que estáo a esquerda devem ficar alinhados à esquerda da div principal e os que estão a direita devem ser alinhados à direita.        
  • Nos campos "Nome" e "Naturalidade" deverá ter como entrada apenas letras e no final elas devem ficar em caixa alta (uppercase);
  • No Campo "RA" deverá ser permitido como entrada apenas seis numeros;
  • No Campo "Data de Nascimento" deverá ter o formato DD/MM/AAAA;
  • Nos campos “Universidade” e “Curso” deve ter um dropdown com pelo menos 3 escolhas e o “Semestre” um dropdown com todos os semestres possíveis (deve ser sinalizado aquele que vocês estão na Unicamp).

5)      Criar um Botão “Submit” (JS)

Ao apertar esse botão deve aparecer um alerta com o campo que não foi preenchido

Ex: “O campo RA não pode ficar em branco”;

Caso todos os campos tenham sido preenchidos corretamente, emita a mensagem: “Formulário submetido com sucesso”.

6)      Ao clicar no botão ele deverá mudar de cinza claro para cinza escuro;

7)      Formatação formulário:

Fonte: Calibri, 16, Negrito; Preto

8)      O formulário deverá mostrar data e hora no canto superior direito

9)      Na lateral direita deverá conter um Menu, conforme a Figura 1. Os nomes em azul são links e deverão ter os seguintes direcionamentos:

    • HOME – deverá atualizar a página;
    • Moodle – https://moodle.ggte.unicamp.br/
    • HTML - https://www.w3schools.com/html/default.asp
    • CSS - https://www.w3schools.com/css/default.asp
    • JS - https://www.w3schools.com/js/default.asp

Interações (JS):

  • Ao passar o ponteiro do mouse por cima de um dos links, ele deve aparecer sublinhado (igual como acontece no moodle);

Ex: HTML --->  HTML   (somente enquanto o ponteiro do mouse estiver sobre o nome).

  • Ao clicar a primeira vez no link, ele deve deixar de ser azul e passar a ser vermelho escuro, sinalizando que você já visitou aquela página.

Ex: HTML  --->  HTML  (somente depois que foi dado o click)

10)   No rodapé, coloque uma linha horizontal, de 2mm e o endereço da FEEC:

Faculdade de Engenharia Elétrica e de Computação 
UNICAMP - Universidade Estadual de Campinas
Av. Albert Einstein 400. 
13.083-852 Campinas - SP, Brazil

Do lado direito, coloque o logo da UNICAMP, alinhado com o texto. 

11)   Salve os arquivos .html, .css e js. São eles que serão entregues!

Estrutura de pasta:

Pasta <RA>

Pasta <IMG>

                Imagens .png

       Pasta <CSS>

                 Arquivo main.CSS

       Pasta <JS>

                 Arquivo main.JS

Arquivo index.html

Obs: Não esqueça de subir para o gitlab!

Lembre-se que seus arquivos devem estar dentro de um diretório com seu RA, dentro do sub-diretório aula3 do repositório delivery. A melhor maneira de você subir seus arquivos é primeiro clonando o repositório delivery na sua máquina, criando um branch de trabalho para você trabalhar e mudando para esse branch. Aí você deve criar o diretório aula3, e depois disso dentro dele o diretório com seu RA. Muito cuidado para não apagar nada dos demais diretórios do repositório, pois qualquer mudança que apagar os demais diretórios na sua máquina, APAGARÃO ESSES ARQUIVOS no repositório central. Você pode, sem querer apagar todos os arquivos de todos seus colegas, que já estejam no repositório. Caso você submeta um merge request que faça isso, o professor irá negar o seu merge request e você terá que refazê-lo. Se por um acaso, o professor não perceber e comitar seu merge request mesmo assim, SERÁ SUA RESPONSABILIDADE reconstruir o repositório. Felizmente isso é sempre possível em um repositório git. Assim, se você fizer uma besteira não intencional, pelo menos essa besteira é consertável. Entretanto, AS VEZES CONSERTAR O REPOSITÓRIO PODE DAR TRABALHO. Tenha cuidado e nunca apague um diretório que não seja seu. Depois que o push tenha sido feito no repositório remoto, aí você pode apagar tudo da sua máquina, se quiser, pois seus dados estarão guardados no repositório central. Mas certifique-se sempre se o upload do seu branch foi feito, antes de apagar seu trabalho local. Após fazer o push do seu branch para o repositório remoto, não se esqueça de fazer um merge request para que seu branch seja incluído no repositório master.
